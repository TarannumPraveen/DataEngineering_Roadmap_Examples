Database Fundamentals Foundation

🗄️ What is a Database?

An organised collection of structured data, stored electronically and managed by a DBMS.

DATABASE: company_db ┌──────────────────────────────────────────────┐ │ TABLE: employees TABLE: departments │ │ ┌──────┬────────┬──────┐ ┌───────┬────────┐│ │ │ id │ name │dept_id│ │dept_id│ name ││ │ ├──────┼────────┼──────┤ ├───────┼────────┤│ │ │ 1 │ Alice │ 10 │ │ 10 │ Eng ││ │ │ 2 │ Bob │ 20 │ │ 20 │ HR ││ │ │ 3 │ Carol │ 10 │ └───────┴────────┘│ │ └──────┴────────┴──────┘ │ │ ↑ Primary Key ↑ Foreign Key │ └──────────────────────────────────────────────┘
Key terms: Table = spreadsheet-like grid · Row = one record · Column = one attribute · Primary Key = unique identifier · Foreign Key = links to another table
📂 SQL Categories (DDL / DML / DQL / TCL / DCL)
SQL Command Categories DDL – Data Definition Language → CREATE ALTER DROP TRUNCATE DML – Data Manipulation Language → INSERT UPDATE DELETE MERGE DQL – Data Query Language → SELECT TCL – Transaction Control Language → COMMIT ROLLBACK SAVEPOINT DCL – Data Control Language → GRANT REVOKE
🏗️ CREATE / INSERT / UPDATE / DELETE 
-- DDL: create the structure
CREATE TABLE employees (
  id         INT          PRIMARY KEY,
  name       VARCHAR(50)  NOT NULL,
  salary     DECIMAL(10,2),
  dept_id    INT
);

-- DML: insert rows
INSERT INTO employees (id, name, salary, dept_id)
VALUES (1, 'Alice', 72000, 10),
       (2, 'Bob',   58000, 20),
       (3, 'Carol', 85000, 10);

-- DML: modify existing data
UPDATE employees
SET    salary = 75000
WHERE  id = 1;

-- DML: remove a row
DELETE FROM employees
WHERE  id = 2;
⚠️ Rule: Always include a WHERE clause in UPDATE / DELETE — omitting it affects every row!
🔍 SELECT / WHERE / ORDER BY / LIMIT 
-- Retrieve all columns
SELECT * FROM employees;

-- Filter rows
SELECT name, salary
FROM   employees
WHERE  salary > 60000
  AND  dept_id = 10;

-- Sort results
SELECT name, salary
FROM   employees
ORDER BY salary DESC;

-- Top N rows
SELECT name, salary
FROM   employees
ORDER BY salary DESC
LIMIT  3;
Result of ORDER BY salary DESC LIMIT 3: ┌─────────┬─────────┐ │ name │ salary │ ├─────────┼─────────┤ │ Carol │ 85000 │ ← row 1 │ Alice │ 75000 │ ← row 2 │ Bob │ 58000 │ ← row 3 └─────────┴─────────┘
📊 Aggregate Functions (COUNT / SUM / AVG / MAX / MIN) 
SELECT
  COUNT(*)                AS total_employees,
  SUM(salary)             AS payroll,
  AVG(salary)             AS avg_salary,
  MAX(salary)             AS top_salary,
  MIN(salary)             AS lowest_salary
FROM employees;
Output: ┌────────────────┬─────────┬────────────┬───────────┬──────────────┐ │total_employees│payroll │avg_salary │top_salary │lowest_salary │ ├────────────────┼─────────┼────────────┼───────────┼──────────────┤ │ 3 │ 218000 │ 72666.67 │ 85000 │ 58000 │ └────────────────┴─────────┴────────────┴───────────┴──────────────┘
📦 GROUP BY & HAVING 
-- Average salary per department
SELECT   dept_id, AVG(salary) AS avg_sal, COUNT(*) AS headcount
FROM     employees
GROUP BY dept_id
HAVING   AVG(salary) > 60000;  -- filter AFTER aggregation
GROUP BY collapses rows sharing the same value: Before GROUP BY: After GROUP BY dept_id: id │ name │ dept_id dept_id │ avg_sal │ headcount ───┼───────┼──────── ────────┼─────────┼────────── 1 │ Alice │ 10 → 10 │ 80000 │ 2 2 │ Bob │ 20 → 3 │ Carol │ 10 WHERE filters rows BEFORE grouping HAVING filters groups AFTER aggregation

🔀 Set Operators (UNION / INTERSECT / EXCEPT) 
-- UNION: combine results, remove duplicates
SELECT id, name FROM employees
UNION
SELECT id, name FROM employees2;

-- UNION ALL: keep duplicates (faster)
SELECT id, name FROM employees
UNION ALL
SELECT id, name FROM employees2;

-- INTERSECT: names appearing in BOTH tables
SELECT name FROM employees_2024
INTERSECT
SELECT name FROM employees_2025;

-- EXCEPT: names in 2024 but NOT in 2025 (left exits)
SELECT name FROM employees_2024
EXCEPT
SELECT name FROM employees_2025;
Rule: Both queries must have the same number of columns with compatible data types.
🔄 Subqueries 
-- Scalar subquery: above-average earners
SELECT name, salary
FROM   employees
WHERE  salary > (SELECT AVG(salary) FROM employees);

-- IN subquery: employees in high-spend departments
SELECT name
FROM   employees
WHERE  dept_id IN (
  SELECT dept_id
  FROM   budget
  WHERE  amount > 500000
);
Execution flow for a scalar subquery: Step 1 → Inner query runs first: SELECT AVG(salary) FROM employees → 71666 Step 2 → Outer query uses that value: SELECT name, salary FROM employees WHERE salary > 71666 Result: Carol (85000), Alice (72000)
↩️ Correlated Subqueries 
-- Each employee vs their own department average
SELECT e.name, e.salary,
       (SELECT AVG(salary)
        FROM   employees x
        WHERE  x.dept_id = e.dept_id) AS dept_avg
FROM employees e;
Correlated vs Regular subquery: Regular: inner query runs ONCE, result reused Correlated: inner query runs ONCE PER OUTER ROW (can be slow on large tables) → prefer CTEs or window functions for performance
📋 CTEs (Common Table Expressions) 
-- Single CTE
WITH dept_avg AS (
  SELECT dept_id, AVG(salary) AS avg_sal
  FROM   employees
  GROUP BY dept_id
)
SELECT e.name, e.salary, d.avg_sal
FROM   employees e
JOIN   dept_avg d ON e.dept_id = d.dept_id
WHERE  e.salary > d.avg_sal;
CTE vs Subquery: CTEs are named, reusable within the query, and much easier to read and debug.
👁️ Views & Temp Tables 
-- VIEW: stored query, no data duplication
CREATE VIEW v_senior_staff AS
  SELECT name, salary, dept_id
  FROM   employees
  WHERE  salary > 80000;

SELECT * FROM v_senior_staff;  -- use like a table

-- TEMP TABLE: materialised, session-scoped
CREATE TEMP TABLE tmp_snapshot AS
  SELECT * FROM employees WHERE dept_id = 10;

-- Good for multi-step transformations
When to use each: VIEW → Reusable logic, always reflects latest data, no extra storage TEMP TABLE → Heavy intermediate result, join it multiple times in same session CTE → Inline, single-query scope, best for readability

Advanced SQL Window Functions

🪟 Window Function Anatomy
Anatomy of a Window Function ROW_NUMBER() OVER ( PARTITION BY dept_id ORDER BY salary DESC ) ──────────── ──── ────────────────────────────────────────────────── Function OVER Window Spec │ ├── PARTITION BY → resets the window per group └── ORDER BY → defines row ordering within window Unlike GROUP BY, window functions do NOT collapse rows — every row is kept!
SELECT
  name, salary,
  ROW_NUMBER() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS row_num,
  RANK()       OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rank,
  DENSE_RANK() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS dense_rank
FROM employees;
ROW_NUMBER vs RANK vs DENSE_RANK (when two salaries tie at 75000): name │ salary │ ROW_NUMBER │ RANK │ DENSE_RANK ──────┼────────┼────────────┼──────┼─────────── Carol │ 85000 │ 1 │ 1 │ 1 Alice │ 75000 │ 2 │ 2 │ 2 Dave │ 75000 │ 3 │ 2 │ 2 ← tie: same rank Bob │ 58000 │ 4 │ 4 │ 3 ← RANK skips 3, DENSE_RANK doesn't
⏪⏩ LAG & LEAD 
SELECT
  sale_date,
  revenue,
  LAG(revenue)  OVER (ORDER BY sale_date)     AS prev_revenue,
  LEAD(revenue) OVER (ORDER BY sale_date)     AS next_revenue,
  revenue - LAG(revenue) OVER (ORDER BY sale_date) AS day_over_day
FROM daily_sales;
LAG looks BACK one row · LEAD looks FORWARD one row: sale_date │ revenue │ prev_rev │ next_rev │ day_over_day ───────────┼─────────┼──────────┼──────────┼───────────── 2026-01-01 │ 1000 │ NULL │ 1200 │ NULL 2026-01-02 │ 1200 │ 1000 │ 950 │ +200 2026-01-03 │ 950 │ 1200 │ 1500 │ -250 2026-01-04 │ 1500 │ 950 │ 1300 │ +550
🌳 Recursive CTE (Hierarchical Data) 
-- Org chart: find all reports under manager id=1
WITH RECURSIVE org_chart AS (
  -- Anchor: start with the root manager
  SELECT id, name, manager_id, 0 AS level
  FROM   employees
  WHERE  id = 1

  UNION ALL

  -- Recursive: find direct reports
  SELECT e.id, e.name, e.manager_id, o.level + 1
  FROM   employees e
  JOIN   org_chart o ON e.manager_id = o.id
)
SELECT REPEAT('  ', level) || name AS hierarchy
FROM   org_chart;
Output — indented hierarchy: CEO (level 0) VP Engineering (level 1) Alice (level 2) Carol (level 2) VP HR (level 1) Bob (level 2)
🔄 Pivot  
-- PIVOT: rows → columns (using conditional aggregation)
SELECT
  dept_id,
  SUM(CASE WHEN year = 2024 THEN revenue END) AS rev_2024,
  SUM(CASE WHEN year = 2025 THEN revenue END) AS rev_2025,
  SUM(CASE WHEN year = 2026 THEN revenue END) AS rev_2026
FROM   sales_by_year
GROUP BY dept_id;
Before PIVOT (tall): After PIVOT (wide): dept │ year │ revenue dept │ 2024 │ 2025 │ 2026 ─────┼──────┼──────── ─────┼───────┼───────┼────── 10 │ 2024 │ 100K → 10 │ 120000 │ 145000 │ 98000 10 │ 2025 │ 120K 20 │ 85000 │ 92000 │ 76000 10 │ 2026 │ 140K

🔄 SCD — Slowly Changing Dimensions
Type 1 — Overwrite (no history kept): Before: id=1 | name=Alice | city=London | is_current=1 After: id=1 | name=Alice | city=Paris | is_current=1 ← London lost Type 2 — Add new row (full history): id │ name │ city │ start_date │ end_date │ is_current ───┼───────┼────────┼────────────┼────────────┼─────────── 1 │ Alice │ London │ 2024-01-01 │ 2026-06-01 │ 0 2 │ Alice │ Paris │ 2026-06-01 │ 9999-12-31 │ 1 ← new row Type 3 — Add column (only previous + current): id │ name │ prev_city │ curr_city ───┼───────┼───────────┼────────── 1 │ Alice │ London │ Paris
-- SCD Type 2 INSERT for a changed record
-- Step 1: expire old row
UPDATE dim_customer
SET    end_date   = CURRENT_DATE,
       is_current = 0
WHERE  customer_id = 101 AND is_current = 1;

-- Step 2: insert new version
INSERT INTO dim_customer
  (customer_id, name, city, start_date, end_date, is_current)
VALUES
  (101, 'Alice', 'Paris', CURRENT_DATE, '9999-12-31', 1);
🧹 Deduplication Pattern 
-- Keep only the latest record per customer
WITH ranked AS (
  SELECT *,
         ROW_NUMBER() OVER (
           PARTITION BY customer_id
           ORDER BY     updated_at DESC
         ) AS rn
  FROM customer_table
)
SELECT * FROM ranked WHERE rn = 1;
Before dedup: After (rn=1 only): cust_id │ updated_at │ city cust_id │ city ────────┼─────────────┼────── ────────┼────── 101 │ 2026-01-01 │ London → 101 │ Paris 101 │ 2026-06-01 │ Paris ← rn=1 102 │ NYC 102 │ 2026-03-01 │ NYC ← rn=1
⚡ Incremental Loads & CDC 
-- Incremental load: only changed records since last run
INSERT INTO warehouse.orders
SELECT *
FROM   source.orders
WHERE  updated_at > (SELECT MAX(updated_at) FROM warehouse.orders);

-- MERGE / UPSERT pattern (most warehouses)
MERGE INTO warehouse.customers AS target
USING      source.customers   AS src
  ON target.customer_id = src.customer_id
WHEN MATCHED AND src.updated_at > target.updated_at THEN
  UPDATE SET target.city = src.city, target.updated_at = src.updated_at
WHEN NOT MATCHED THEN
  INSERT (customer_id, city, updated_at)
  VALUES (src.customer_id, src.city, src.updated_at);
Full Load vs Incremental Load: Full Load → truncate + reload everything (simple, slow, expensive) Incremental → only new/changed rows (faster, needs watermark column) CDC → stream every INSERT/UPDATE/DELETE from transaction log
✅ Data Quality Checks 
-- Null check
SELECT COUNT(*) AS null_emails
FROM   customers
WHERE  email IS NULL;

-- Duplicate check
SELECT customer_id, COUNT(*) AS cnt
FROM   customers
GROUP BY customer_id
HAVING COUNT(*) > 1;

-- Referential integrity check
SELECT o.order_id
FROM   orders o
LEFT JOIN customers c ON o.customer_id = c.customer_id
WHERE  c.customer_id IS NULL;   -- orphan orders

-- Range / freshness check
SELECT MAX(updated_at) AS last_update,
       DATEDIFF(NOW(), MAX(updated_at)) AS days_stale
FROM   orders;

⚡ Performance Tuning Optimization

🗂️ Indexes & Execution Plans 
-- Create an index on a frequently filtered column
CREATE INDEX idx_emp_dept ON employees(dept_id);
CREATE INDEX idx_emp_sal  ON employees(salary DESC);

-- Check the execution plan
EXPLAIN ANALYZE
SELECT * FROM employees WHERE dept_id = 10;
Query optimisation checklist: ✓ Avoid SELECT * → list only the columns you need ✓ Filter early → push WHERE conditions as close to the source as possible ✓ Use indexes → on JOIN keys, WHERE columns, ORDER BY columns ✓ Avoid functions → in WHERE clauses (prevents index use) ✓ Partition tables → for very large tables, filter by partition key ✓ Check EXPLAIN → look for Seq Scan → Index Scan improvements
